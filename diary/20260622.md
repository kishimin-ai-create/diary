## Session 4 — Backend One-Line Bug Fixes (F5 & F8)

Two small but meaningful one-line fixes were applied to the backend and landed in a single commit (`924b422`).

### Fix A (F8) — `node:crypto` import prefix in test file

**File:** `backend/src/services/auth.service.small.test.ts` (line 14)

```diff
- import { scryptSync, randomBytes } from "crypto";
+ import { randomBytes, scryptSync } from "node:crypto";
```

The bare `"crypto"` specifier is technically resolvable, but it can be silently shadowed by an npm package of the same name. The `node:` prefix is the modern Node.js convention and was already in use throughout the production code. Aligning the test import removes the inconsistency and eliminates the shadowing risk entirely.

As a side effect, ESLint's `simple-import-sort` plugin reordered the named imports to alphabetical order (`randomBytes` before `scryptSync`), which is the expected auto-fix behaviour.

### Fix B (F5) — Empty-string password bypass in `loginSchema`

**File:** `backend/src/controllers/auth.controller.ts` (line 21)

```diff
- password: z.string().max(255),
+ password: z.string().min(1).max(255),
```

Without `min(1)`, an empty-string password passed Zod validation and continued down the happy path until `scryptSync` ran a full key-derivation operation before the 401 was finally returned. Adding `min(1)` rejects the request at the validation layer with a 400 instead, sparing the CPU from an unnecessary scrypt call on obviously invalid input. This keeps validation tight and consistent with the principle of failing fast.

## Session 5 — Bug Fix: Widen Newline Regex in `generateContentPreview`

**Commit:** `7eb6024`  
**Files changed:** `backend/src/models/diary.ts`, `backend/src/models/diary.small.test.ts`

### Background

The `generateContentPreview` function in `backend/src/models/diary.ts` is responsible for turning a raw diary entry into a short text snippet — replacing newlines with spaces and truncating to 100 characters. The function was working correctly for Unix content, but a subtle regex gap left it vulnerable to two other common line-ending styles.

### Root Cause

The original implementation used `/\n/g` as the replacement pattern:

```ts
const processed = content.replace(/\n/g, " ");
```

This only matches Unix LF (`\n`). Two other line-ending conventions were silently ignored:

- **Windows CRLF (`\r\n`)** — the `\r` character was left in the output, resulting in a stray carriage-return character in the preview string.
- **Old Mac CR (`\r`)** — not matched at all; the line break was preserved as-is instead of being replaced with a space.

Both cases would produce corrupted preview text whenever content originated from a Windows editor or an old Mac tool, which is a realistic scenario for a diary application.

### Fix Applied

The regex was updated to a single alternation that covers all three conventions:

```ts
// Before
const processed = content.replace(/\n/g, " ");

// After
const processed = content.replace(/\r?\n|\r/g, " ");
```

How the new regex works:

| Pattern branch | Matches |
|---|---|
| `\r?\n` | Unix LF (`\n`) and Windows CRLF (`\r\n`) |
| `\r` | Old Mac standalone CR (`\r`) |

The alternation evaluates left to right, so `\r\n` is consumed as a single unit by the first branch, preventing a double-space from being emitted.

### Tests Added

A new `describe("Line Ending Normalisation", ...)` block was added to `backend/src/models/diary.small.test.ts` with three focused test cases:

1. **Windows CRLF** — `"line one\r\nline two"` → `"line one line two"`
2. **Old Mac CR** — `"line one\rline two"` → `"line one line two"`
3. **Mixed line endings** — `"a\nb\r\nc\rd"` → `"a b c d"` (verifies all three styles in one pass)

### Verification

| Check | Result |
|---|---|
| TypeScript compile | ✅ clean |
| Test suite (`bun test`) | ✅ 85 pass, 0 fail |

The three new test cases bring the total from 82 to 85. All existing tests continued to pass without modification.

### Notes

- The fix is a pure regex change — no logic restructuring, no interface change.
- The truncation and ellipsis behaviour of `generateContentPreview` is unaffected.
- No pending items; the fix is complete and committed.

## Session 6 — Backend API Persistence and Runtime OpenAPI

The backend API was wired from the testable Hono application factory into a production-style runtime entrypoint backed by Drizzle and MySQL.

### What changed

The work connected `backend/src/index.ts` to `createApp`, loaded runtime configuration from environment variables, and instantiated concrete Drizzle repositories for users and diary entries. The service layer stayed behind the existing repository interfaces, so the integration tests can still inject mock repositories while production code uses the database-backed implementations.

### OpenAPI

The backend OpenAPI contract was changed from a static YAML file to runtime generation through `hono-openapi`.

The static `docs/spec/backend/openapi.yaml` file was removed. Instead, `backend/src/app.ts` now exposes `/openapi.json`, and the auth and diary controllers attach route metadata with `describeRoute` and Zod response resolvers.

This keeps the OpenAPI description closer to the route implementation and avoids maintaining a separate handwritten contract file for the same API.
