## Summary

The overall design is solid: Clean Architecture layers are respected, Zod validates at every API boundary, ServiceError propagates cleanly through the error-handler chain, and the test suite is thorough and well-structured. Two critical defects require immediate attention before deployment: a timing side-channel in `login` that undermines the anti-enumeration guarantee despite identical error messages, and the backend entry point (`index.ts`) never being wired to `createApp`, meaning the entire feature is dead in production. Three additional findings cover a DoS-enabling missing validation cap, a fragile integration-test fixture, and minor gaps in line-ending normalization and login input validation.

Timing Side-Channel in `login` Leaks Email Existence**

**File**: `backend/src/services/auth.service.ts`
**Lines**: L56–L63
**Issue**:
When the email is not found, `login` throws immediately at L58 — no `scryptSync` is ever called (~0 ms). When the email *is* found but the password is wrong, `verifyPassword` calls `scryptSync` at L60, which is intentionally slow (~50–100 ms). An attacker can distinguish the two cases purely by measuring response time, even though both return `"Invalid email or password."`. This directly defeats the user-enumeration protection the test suite verifies at the message level.

```ts
// Current — fast path reveals email absence
const user = await this.userRepo.findByEmail(input.email);
if (!user) {
  throw createError(401, "Invalid email or password."); // returns in <1 ms
}
const valid = verifyPassword(input.password, user.passwordHash); // ~80 ms
```

**Recommendation**:
Always run `verifyPassword` regardless of whether the user was found, then branch on the combined result:

```ts
// Pre-computed at module load (one-time cost)
const DUMMY_HASH = hashPassword("__dummy__");

async login(input) {
  const user = await this.userRepo.findByEmail(input.email);
  const storedHash = user?.passwordHash ?? DUMMY_HASH;
  const valid = verifyPassword(input.password, storedHash);
  if (!user || !valid) {
    throw createError(401, "Invalid email or password.");
  }
  // issue JWT …
}
```

This keeps scrypt time constant regardless of whether the email exists.

`index.ts` Not Wired to `createApp` — Entire Feature Dead in Production**

**File**: `backend/src/index.ts`
**Lines**: L1–L9
**Issue**:
The application entry point still contains the scaffold boilerplate that responds `"Hello Hono!"` to every request. It never imports `createApp` and never mounts auth or diary routes. Running the backend as deployed (`bun run src/index.ts` or any production start command) would serve none of the new API — all endpoints return 404 or the placeholder text. This has no test coverage because the integration tests import `createApp` directly and never exercise `index.ts`.

```ts
// Current state of index.ts
import { Hono } from 'hono'
const app = new Hono()
app.get('/', (c) => c.text('Hello Hono!'))
export default app
```

**Recommendation**:
Replace the placeholder with a real wiring that reads secrets from environment variables:

```ts
import { createApp } from "./app";
import { SqliteUserRepository } from "./infrastructure/user.repository.sqlite";
import { SqliteDiaryRepository } from "./infrastructure/diary.repository.sqlite";

const jwtSecret = process.env["JWT_SECRET"];
if (!jwtSecret) throw new Error("JWT_SECRET env var is required");

const app = createApp({
  userRepo: new SqliteUserRepository(),
  diaryRepo: new SqliteDiaryRepository(),
  jwtSecret,
});

export default app;
```

The exact repository implementations will depend on the chosen DB adapter, but the pattern is clear. A smoke test that hits `GET /api/diaries` against the real `index.ts` export would prevent this regression.

`pageSize` Has No Upper Bound — Potential DoS via Unbounded Queries**

**File**: `backend/src/controllers/diary.controller.ts`
**Lines**: L17–L18
**Issue**:
`pageSize: z.coerce.number().int().min(1)` has no `.max()`. Any caller — including an unauthenticated one — can issue `GET /api/diaries?pageSize=1000000`, which is forwarded verbatim to the repository `findMany` call. A real SQL implementation would execute `LIMIT 1000000`, scanning and materialising a potentially huge result set in memory. Even against the mock repo in tests, the missing cap means the validation contract is incomplete.

**Recommendation**:
Add a reasonable cap and document it. A common choice for a diary-style app is 100:

```ts
pageSize: z.coerce.number().int().min(1).max(100).default(10),
```

Also add a corresponding test case:
```ts
test("returns 400 when pageSize exceeds 100", async () => { … });
```

Login Happy-Path Integration Test Uses Hardcoded Scrypt Hash**

**File**: `backend/tests/integrations/auth.medium.test.ts`
**Lines**: L243–L246
**Issue**:
The `POST /api/auth/login` happy-path test injects a literal scrypt digest string rather than calling `hashPassword("Password123")`:

```ts
passwordHash: "798c0534f33660b2a73fc6f65f03e943:00b3916a7ed18d96…",
```

This test is supposed to verify the end-to-end round-trip (`hashPassword` → store → `verifyPassword`). Instead it only verifies that `verifyPassword` works on a pre-baked hash. If the key length, salt encoding, or output format in `hashPassword` ever changes, this test will silently return a 401 (false negative) rather than failing loudly, masking a real regression.

**Recommendation**:
Import `hashPassword` and compute the fixture hash at test setup time:

```ts
import { hashPassword } from "../../src/models/user";

// Inside the test:
findByEmail: mock(() =>
  Promise.resolve({
    ...TEST_USER,
    passwordHash: hashPassword("Password123"),
  })
),
```

This makes the integration test genuinely exercise the full hash round-trip.

`loginSchema` Missing Minimum Length on `password` — Unnecessary scrypt Invocation**

**File**: `backend/src/controllers/auth.controller.ts`
**Lines**: L19–L22
**Issue**:
`loginSchema` validates `password: z.string().max(255)` with no lower bound. An empty string (`""`) passes validation and flows into `authService.login`, which calls `verifyPassword("", hash)` and triggers a full `scryptSync` computation before returning 401. While this cannot cause a security bypass, it creates an avenue for cheap CPU cost with zero useful work.

Compare `registerSchema` (same file, L10–L17) which does enforce `min(8)` and character-class refinements.

**Recommendation**:
Add `.min(1)` to match the intent that a login attempt with no password is malformed input, not a failed auth attempt:

```ts
const loginSchema = z.object({
  email: z.string().email(),
  password: z.string().min(1).max(255),
});
```

**<sub><sub>![P3 Badge](https://img.shields.io/badge/P3-blue?style=flat)</sub></sub>
`generateContentPreview` Does Not Normalise `\r\n` or `\r` Line Endings**

**File**: `backend/src/models/diary.ts`
**Lines**: L6
**Issue**:
`content.replace(/\n/g, " ")` replaces Unix line feeds only. Content pasted from Windows editors contains `\r\n` sequences; the `\r` is left in place and appears as an invisible control character in the preview. Old macOS content uses bare `\r`, which is not touched at all. Neither edge case is covered by the current tests.

**Recommendation**:
Widen the regex to cover all three standard newline styles:

```ts
const processed = content.replace(/\r?\n|\r/g, " ");
```

And add a test case:
```ts
test("replaces Windows CRLF (\\r\\n) with a space", () => {
  expect(generateContentPreview("line one\r\nline two")).toBe("line one line two");
});
```

**<sub><sub>![P3 Badge](https://img.shields.io/badge/P3-blue?style=flat)</sub></sub>
`scryptSync` Blocks the Event Loop**

**File**: `backend/src/models/user.ts`
**Lines**: L10, L27
**Issue**:
`scryptSync` is CPU-intensive by design and runs synchronously on the main thread. Under concurrent load — even on a single-admin app — every register or login request blocks all other I/O for the duration of the hash computation (~50–100 ms at the default `N=16384`). This degrades overall API latency for any requests that arrive during that window.

**Recommendation**:
Switch to the async variant using `util.promisify`:

```ts
import { promisify } from "node:util";
import { randomBytes, scrypt, timingSafeEqual } from "node:crypto";

const scryptAsync = promisify(scrypt);

export async function hashPassword(plain: string): Promise<string> {
  const salt = randomBytes(16).toString("hex");
  const hash = (await scryptAsync(plain, salt, 64) as Buffer).toString("hex");
  return `${salt}:${hash}`;
}
```

Alternatively, Bun provides `Bun.password.hash` / `Bun.password.verify` which use bcrypt or argon2 natively and are always async.

**<sub><sub>![P3 Badge](https://img.shields.io/badge/P3-blue?style=flat)</sub></sub>
`import from "crypto"` vs `"node:crypto"` Inconsistency in Test File**

**File**: `backend/src/services/auth.service.small.test.ts`
**Lines**: L14
**Issue**:
The test file imports `{ scryptSync, randomBytes }` from `"crypto"` (bare specifier) while the implementation (`models/user.ts`) uses the modern `"node:crypto"` form. Both resolve to the same built-in module today, but the bare `"crypto"` specifier can be shadowed by an npm package of the same name. Using the explicit `node:` prefix is the current Node.js / Bun convention and prevents that ambiguity.

**Recommendation**:
```ts
import { scryptSync, randomBytes } from "node:crypto";
```

## Conclusion

**Verdict**: ⛔ Fail — Pass with Conditions

The two P1 findings must be resolved before this branch can merge:

1. **Fix the timing side-channel** in `AuthService.login` by always calling `verifyPassword` with either the real or a dummy hash.
2. **Wire `index.ts` to `createApp`** so the backend actually serves the new API when deployed.

The two P2 findings are strongly recommended for the same release:

3. **Add `.max(100)` to `pageSize`** in `listQuerySchema`.
4. **Replace the hardcoded scrypt hash** in `auth.medium.test.ts` with a `hashPassword()` call.

The P3 findings (empty-password validation, `\r\n` normalisation, async scrypt, import style) can be addressed in a follow-up without blocking the release.
